会议主题：混合线性模型推理架构与调度策略分享
发言人：周等等、PadawanZH、Zhipeng
会议摘要：本次分享探讨了基于混合线性模型的推理架构，通过 Prefill as a Service 实现异构硬件部署，利用数学建模优化调度策略，并针对 DeepSeek V4 的稀疏 KV Cache 进行了适用性评估。

------------------------------------------------------------------------

一、混合线性模型驱动的推理架构变革

针对传统稠密模型在长序列推理中 KV Cache 传输带宽爆炸的问题，会议提出了基于混合线性模型（如 Mamba、RWKV）的新架构方案：

1. 带宽瓶颈的突破

-   KV Cache 传输量显著降低：混合线性模型（如 Mamba）的 State 矩阵大小固定，不随序列长度线性增长，解决了长序列下 KV Cache 传输带宽过高的问题。
-   远程部署可行性验证：经测算，512 张 H200 在 32k 序列长度下，混合模型仅需 170 Gbps 带宽，可在远端机房部署；而传统稠密模型（如 MiniMax M2.5）需 3.8 Tbps，仅能同机房部署。

2. 异构硬件集群设计

-   计算与存储分离：系统分为远端 Prefill Service（高算力密集型，如 H200）和本地 Decode Cluster（高带宽推理型，如 H20），利用 VPC 网络连接。
-   网络分层架构：本地 Decode 集群内部使用 RDMA（800 Gbps）保证低时延，集群间通过普通 VPC（100 Gbps）传输 KV Cache，降低网络成本。

二、调度策略与数学模型优化

会议详细拆解了调度器的核心逻辑，并建立了基于资源利用率的数学模型：

1. 双时间尺度调度机制

-   短时调度（Short-term）：基于启发式策略，根据当前带宽状况和 KV Cache 命中率决定请求路由。带宽稀缺时优先本地计算，带宽丰富时拉取远端 KV Cache。
-   长时调度（Long-term）：定期分析系统瓶颈（Prefill 或 Decode），动态调整 Prefill 实例（P）与 Decode 实例（D）的比例，以最大化资源利用率。

2. 最优参数求解

-   理想状态建模：建立方程组，假设 Prefill 计算速率、传输带宽速率与 Decode 计算速率三者完美匹配（Exact Match），此时系统吞吐量最优。
-   关键参数确定：通过 Profiling 反解方程，确定最优调度阈值 T（19.4k）及实例配比（P:D=3:5），实现 50% 请求卸载至远端。

三、性能评估与成本收益

1. 吞吐量与延迟提升

-   吞吐量显著增长：相比同构基线（96 张 H20），异构部署方案（32 张 H200 + 64 张 H20）吞吐量提升 54%，相比单纯异构部署提升 32%。
-   响应时间优化：Prefill 响应时间（TTFT）降低了 50%-64%，系统整体延迟得到改善。

2. 成本效益分析

-   带宽利用率极低：系统出站负载仅 13 Gbps，占 100 Gbps 总带宽的 43%，商用以太网即可满足需求且有充足余量。
-   性价比优势：尽管 H200 单卡成本更高，但折算吞吐量收益后，整体性价比仍提升了 15%。

四、自由交流与答疑环节

1. 关于调度算法的动态性探讨

-   【提问人】PadawanZH：调度算法是否考虑了 Prefill 执行时间较长导致带宽状态变化的情况？
-   【回答人】周等等：当前论文中的算法是静态的，基于 Profiling 预设参数，未做动态调整。在实际生产环境中，需要引入动态扩缩容和实时反馈机制。

2. 关于 DeepSeek V4 的适用性探讨

-   【提问人】Zhipeng：DeepSeek V4 采用了稀疏 KV Cache 技术（如 Sliding Window Attention），是否会影响该架构的适用性？
-   【回答人】周等等：如果 DeepSeek V4 仅在计算时节省算力，但仍需保存完整的 KV Cache 进行传输，则不适用。若其传输的 KV Cache 本身就是稀疏压缩后的结果（如 128 倍压缩），则适用。
